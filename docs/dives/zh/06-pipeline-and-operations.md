# 流水线与运维：从单轮到持续自主

> 前置阅读：[双通道机制](03-dual-channel-mechanism.md)、[LLM 元控制器](04-llm-meta-controller.md)、[搜索引擎](05-search-engine.md)。

## 单轮：10 步流水线

每个优化轮次遵循固定的 10 步流水线。逻辑是：**反思发生了什么 → 校准框架 → 搜索更好的参数 → 部署**。

### 总览

```
阶段 1：线上更新（步骤 1-10）
    反思线上反馈，校准双通道，准备搜索框架

阶段 2：离线搜索
    Optuna TPE 在校准后的框架内找最优参数

阶段 3：发布
    参数发布到线上配置通道，写入交接状态供下一轮使用
```

两个可选人工检查点：
- **检查点 1**（步骤 8 后）：审阅 LLM 提案后再进入搜索
- **检查点 2**（搜索后）：审阅搜参结果后再发布到线上配置通道

自主模式（`--yes`）下两个检查点自动通过。

### 逐步分解

**步骤 1-4：数据收集**

| 步骤 | 动作 | 输出 |
|------|------|------|
| 1 | 拉取当前时间窗口的 A/B 实验报告 | 原始指标 delta |
| 2 | 首次运行时初始化 Memory DB | 持久化状态库 |
| 3 | 记录离线数据：上轮搜参结果和预测指标 | 新 episode 记录（离线字段填充） |
| 4 | 记录线上数据：解析 A/B delta 并附加到 episode | episode 记录补全线上结果 |

**步骤 5：LMS 信念更新**

LMS 算法读取最新的（离线预测, 线上实际）配对，更新全部 6 条迁移模型的斜率和截距。无条件执行，每轮修正 20% 的残差。

**步骤 6-8：LLM 元控制器**

| 步骤 | 动作 |
|------|------|
| 6 | 从 Memory DB 导出 LLM 上下文 JSON（20 轮 episodes + 30 条更新记录 + 6 条先验关系 + 惩罚历史） |
| 7 | 调用 LLM API，接收含 delta_intercept 和 penalty_multiplier 的 JSON 提案 |
| 8 | 将截距调整写入 Memory DB；缩放下一轮配置中的惩罚权重 |

如果 LLM 返回空提案，系统不做框架调整继续运行。

**步骤 9-10：算法精调**

| 步骤 | 动作 |
|------|------|
| 9 | 用更新后的迁移模型推导 target_range：什么样的离线指标范围对应可接受的线上结果 |
| 10 | 根据各约束在线上是否被满足，应用偏好通道的非对称乘性更新规则 |

步骤 10 完成后，完整的搜索配置就绪：target_range 定义可行域，penalty_weight 定义违约代价。

## 三种运行模式

10 步流水线是构建块。三种模式决定它如何被调用：

### 模式 1：Coldstart（引导启动）

**时机**：全新实验，没有任何历史——无 Memory DB、无先前结果、无 Handoff 状态。

**做什么**：
1. 将一组已知可用的初始参数发布到线上配置通道
2. 将时间窗口起点锚定到下一个 15 分钟边界
3. 写入初始状态快照

**不做什么**：不创建 Memory DB、不调用 LLM、不搜参。只是让参数上线，让实验开始积累 A/B 数据。

### 模式 2：Initial（首次完整运行）

**时机**：实验已在运行并积累了足够的 A/B 数据（通常 ≥ 3.5 小时），但流水线从未执行过。

**做什么**：
1. 执行完整的 10 步流水线
2. 创建并初始化 Memory DB
3. 建立供下一轮读取的交接状态快照
4. 在两个检查点暂停等待人工审阅（除非指定 `--yes`）

### 模式 3：YOLO（持续自主循环）

**时机**：基线已建立，系统进入全自主运行。

```
while true:
    等待（距上次推送 ≥ 30 分钟 且 数据窗口 ≥ 3.5 小时）
    对齐时间窗口到 15 分钟边界
    尝试拉取 A/B 数据（最多重试 5 次，每次间隔 5 分钟）
    执行完整 10 步流水线（--yes 自动确认）
    更新状态快照
    追加审计日志
```

设计目标：
- **无人值守**：所有确认自动通过
- **可恢复**：状态持久化到快照；崩溃重启后从上次完成的轮次继续
- **自愈**：A/B 拉取失败自动重试；非 A/B 错误立即退出
- **可审计**：每轮产出详细的审计日志条目

### 模式进阶

```
Coldstart -----> Initial -----> YOLO
  (引导)         (首次运行)     (无限循环)
    |               |              |
    v               v              v
  初始参数发布   持久化状态      持续自我改进
  + 时间锚点    + 交接快照
```

每个模式的输出是下一个模式的前置条件。前置条件已满足时可以跳过前面的阶段。

## 时间窗口管理

**15 分钟对齐**：所有窗口边界对齐到 15 分钟刻度（14:00、14:15、14:30...），匹配 A/B 平台的报告粒度。

**自动衔接**：每轮的窗口从上一轮的终点开始。第 N 轮覆盖 10:00-13:30，第 N+1 轮从 13:30 开始。

**最小积累**：数据窗口至少 3.5 小时，作为提高信号稳定性的操作阈值；是否足够仍取决于样本量、方差和效应大小。

**最小间隔**：线上发布之间至少 30 分钟，防止变更速度超过 A/B 平台的测量能力。

## 状态快照

跨重启持久化自主循环状态：

```
round=<N>
last_push_time=<timestamp>
last_online_from=<timestamp>
last_online_to=<timestamp>
prev_params_snapshot={...}
```

重启时读取最近一次快照，从上一个完整窗口继续。

## 审计日志：飞行记录仪

每轮自主循环都会追加一条结构化审计条目，记录：部署的参数摘要、A/B 结果摘要、LLM 提案摘要和新参数摘要。这是全自主运行的审计轨迹——运维人员可以定期审阅以监控趋势、诊断问题、验证系统决策的合理性。

## 容错处理

| 故障类型 | 处理方式 |
|----------|----------|
| A/B 数据拉取失败 | 重试最多 5 次，每次间隔 5 分钟。全部失败则等下一轮 |
| 非 A/B 故障（代码错误、DB 损坏） | 立即退出，诊断后手动重启 |
| LLM API 故障 | 回退到空提案（本轮不做框架调整） |
| Optuna 崩溃 | 从搜索后端的检查点恢复 |
| 进程被杀/崩溃 | 重启时读取最近状态快照，从上次完成的轮次继续 |

设计哲学：**A/B 数据不可用是暂时的，值得重试。其他故障是结构性的，不应盲目重试。**
