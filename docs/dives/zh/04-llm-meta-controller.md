# LLM 元控制器

> 前置阅读：[理论基础](01-theoretical-foundations.md)、[双通道机制](03-dual-channel-mechanism.md)。

## LLM 做什么，不做什么

LLM 位于三层架构的第二层——人工定义的目标（第一层）和数值参数搜索（第三层）之间。它的职责被精确界定：

- 它**不**搜索排序参数——那是 Optuna 的工作。
- 它**不**定义业务目标——那是人工的工作。
- 它**只**基于从历史数据中识别的规律，调整搜索运行时的框架。

这就是"元控制器"的含义：它控制的是控制器。它不直接优化参数，而是优化*参数被优化的条件*。

### 教练-运动员类比

LLM 是一个审看比赛录像的教练。它查看最近 20 场比赛（episodes）的表现和所有战术调整（30 条校准更新）的轨迹。基于识别的规律——"最近 5 场我们防线太松了"——它调整下一场比赛的战术框架。

搜索引擎是运动员。给定战术框架，它通过大规模试验和评估为当前对手（数据集）找到最佳执行方案。运动员不需要分析录像；教练不需要跑百米冲刺。

## 两个正交旋钮

LLM 的全部输出是一个 JSON 提案，包含两个维度的调整：

### 旋钮 1：`delta_intercept` — 信念修正

- 范围：[-0.1, +0.1]
- 作用对象：离线-线上迁移函数的截距
- 效果：平移 `target_range`——搜索空间中约束边界的位置

正值："当前映射偏悲观，线上实际比预测好，上调截距。"
负值："当前映射偏乐观，线上实际比预测差，下调截距。"
零值："证据不足或噪声太大，不动。"

### 旋钮 2：`penalty_multiplier` — 偏好修正

- 范围：[0.5, 2.0]
- 作用对象：缩放特定约束的惩罚权重
- 效果：改变搜索对违约的惩罚力度

大于 1.0："这条约束被反复违反，加大惩罚，让搜索更保守。"
小于 1.0："这条约束卡得太紧了，浪费了优化空间，放松一点。"
等于 1.0："不需要调整。"

### 为什么只有这两个旋钮？

它们直接对应 SEU 理论中的两种误差类型。LLM 无法直接提议底层搜参参数的取值，因为那些值依赖当前数据集的特征分布——仅凭推理无法确定，需要在实际数据上做数值搜索。

框架参数则编码**跨轮次的结构性知识**：例如"离线对 GMV 的预测连续几轮偏乐观"或"某条订单约束持续挡不住违约"。这恰好是 LLM 擅长识别的规律类型。

## LLM 看到什么：上下文结构

LLM 接收一个结构化 JSON 上下文，由流水线第 6 步从 Memory DB 中组装：

### 先验关系表

系统维护着若干条离线-线上迁移关系，例如：

- 离线 GMV 影响力 → 线上 GMV
- 离线订单影响力 → 线上订单
- 离线广告影响力 → 线上广告收入 / 展示 / 点击
- 离线广告主价值代理指标 → 线上广告主价值

LLM 看到的是这些关系当前的 slope / intercept 估计及其变化轨迹，而不是一组固定常量。

### 最近 Episodes（最多 20 轮）

每个 episode 包含一轮完整的离线-线上配对：离线搜参结果（最优参数、预测指标、约束范围、惩罚权重）和线上 A/B 结果（各业务指标的实际 uplift）。

LLM 可以跨多轮对比预测与实际，识别系统性偏差。

### 更新历史（最多 30 条）

所有 LMS 和 LLM 对斜率/截距的修正轨迹，以及惩罚权重的变化轨迹。让 LLM 判断修正是在收敛（系统趋稳）还是在发散（系统失控）。

## 一个示意提案的逐条解读

下面是一段经过抽象化的示意提案：

```json
{
  "proposals": [
    {
      "relation_key": "primary_gmv_relation",
      "delta_intercept": -0.01,
      "penalty_multiplier": 1.10,
      "reason": "最近多轮主目标在线结果持续弱于离线预测，且约束已多次收紧，说明当前迁移关系仍偏乐观。",
      "evidence_episode_keys": ["ep_a", "ep_b", "ep_c", "ep_d"]
    }
  ]
}
```

解读：
- **delta_intercept = -0.01**：小幅下调。LLM 判断离线到线上的映射仍略偏乐观，因此先做温和修正，而不是一次大跳变。
- **penalty_multiplier = 1.10**：略微加大主约束的惩罚。在信念修正之外，让搜索再保守一点，形成双重对冲。
- **evidence_episode_keys**：显式引用若干 episode。提示词要求每个提案都附证据，迫使 LLM 将推理锚定在具体观测上，而非生成看似合理但无根据的说辞。

## LLM 与 LMS 的协作

每轮内，两个机制依次更新截距：

```
Step 5 (LMS):  intercept += eta × 误差              // 必定执行，小步
Step 8 (LLM):  intercept += delta_intercept          // 可能执行，可能大步

最终截距 = 初始值 + LMS 修正 + LLM 修正
```

互补优势：

| 场景 | 仅 LMS | 仅 LLM | LMS + LLM |
|------|--------|--------|-----------|
| 缓慢漂移 | 处理良好 | 大材小用 | LMS 处理；LLM 安静 |
| 突变 | 约 15 轮收敛 | 1-2 轮跳到位 | LLM 跳变；LMS 微调 |
| 高噪声 | 过度修正（20% 的噪声也修） | 可选择不动 | LMS 小幅移动；LLM 按兵不动 |

核心洞察：**LLM 不是"更好的算法"——它是一个知道什么时候不该碰操纵杆的飞行员。**

## 安全架构

### 幅度限制

无论 LLM 多有信心，修正都被钳制：

| 旋钮 | 单轮最大影响 |
|------|-------------|
| `delta_intercept` | ±0.1（即使 LLM 认为截距偏了 0.5） |
| `penalty_multiplier` | 0.5x 到 2.0x（惩罚每轮最多翻倍或减半） |

即使 LLM 完全判断错误，也无法在单轮内造成灾难性偏移。下一轮有机会纠正。

### 空提案兜底

如果 LLM 返回空提案（判断证据不足），流水线以保守默认值继续运行——不调整框架，而非编造一个调整。

### 信念大更新后冻结偏好

当信念通道做了大幅调整时，偏好通道冻结 1 轮。否则下一轮的 A/B 结果同时反映两个变化，无法归因。冻结保持信号清晰。

## 更深层的原因：什么算"经验"？

LLM 调框架参数而非底层参数，根本原因在于一个关于知识的问题：

**第 N 轮的什么观察，在第 N+1 轮仍然有用？**

底层参数最优值是**数据依赖的一次性事实**。"某个底层乘子在这批数据上的最优值是多少"——换一批数据，最优值就可能变。这不是经验，是档案。

框架参数编码的是**跨轮次的结构性规律**。"离线 GMV 预测连续几轮偏乐观"——这告诉你环境的某个特征很可能在下一轮继续存在。这是经验：可迁移的知识。

LLM 处理经验。搜索引擎处理数据。让 LLM 去选参数值，就像让教练去跑百米；让搜索引擎去诊断结构性趋势，就像让短跑运动员去设计训练计划。这种分工不是妥协——而是对两类根本不同问题类型的尊重。
